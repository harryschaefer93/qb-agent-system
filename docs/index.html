How it works

From a free-form ask to a validated, documented, pushed POC — in four moves.

1

Scope → Brief

scoper intakes a free-form request, researches the customer, and produces the shared BRIEF.md every agent reads.

2

QB orchestrates

QB classifies the task into one of seven types, then stops at two mandatory checkpoints before any work begins.

3

Specialists build

ARCH, DEV, INFRA, QA, DIAGRAM, DOCS and REPO fan out, with quality gates and bounce-on-fail loops.

4

Retro improves

retro mines session history, scores the fleet, and files tracked improvements that evals then measure.

Human-in-the-loop by design. QB never writes code itself — it classifies, gates, and routes. Both checkpoints are enforced through structured questions; chat text never counts as approval.

Get started

These are agent definitions, not a standalone app — they run inside GitHub Copilot (VS Code Copilot Chat and/or the Copilot CLI).

1 · Install the agents

# Clone the repo
git clone https://github.com/harryschaefer93/qb-agent-system
cd qb-agent-system

# Copilot CLI: copy CLI agents into ~/.copilot/agents/
#   scoper.md, retro.md

# VS Code Copilot Chat: copy *.agent.md into your
# custom-agents dir, then pick QB in the agent selector

2 · Run the pipeline

# (Optional) drop a BRIEF.md at your workspace root,
# or let scoper generate one.

# Invoke QB with a request, e.g.
QB ▸ "Build a POC for a RAG chatbot on Azure."

# QB confirms scope (Checkpoint 1), runs QA,
# presents a plan (Checkpoint 2), then routes work.

3 · Run the eval harness (optional)

cd evals
pip install -e .

# Dry-run an agent's behavioral test cases
python -m runner.cli run-behavioral qb --dry-run

# Mine session telemetry for IMP evidence
python -m runner.telemetry scan --since 30d

Adapt before you run

  • Model IDs — each agent's model: field points at claude-opus-4.6-1m; swap for a model your license exposes.
  • ~/.copilot paths — retro and parts of evals assume the standard local Copilot session store (OS-agnostic).
  • MCP servers — any you don't have are ignored; trim the tools: list if you prefer.
  • FDPO conventions — Entra ID + RBAC, no API keys, disableLocalAuth: true. No change needed for FDPO tenants.
Note on examples. All customer names and identifiers in this repo (Contoso, Fabrikam, Woodgrove, Adatum…) are fictional placeholders. Nothing here contains real customer data.
