How it works

From a free-form ask to a validated, documented, pushed POC — in four moves.

1

Scope → Brief

scoper intakes a free-form request, researches the customer, and produces the shared BRIEF.md every agent reads.

2

QB orchestrates

QB classifies the task, sends SCOUT on cheap read-only recon, and stops at two human checkpoints — CHECKPOINT 2 presents a design preview before any implementation. A session autonomy dial (guided · standard · trusted) tunes everything except the hard gates.

3

Specialists build

ARCH, DEV, INFRA, QA, DIAGRAM, DOCS and REPO fan out — multi-track work runs in isolated git worktrees with attributed merges — behind quality gates and bounce-on-fail loops. ORACLE gives a cross-family second opinion at conflicts and risky approvals.

4

Retro improves

retro mines session history and files tracked improvements; the imp agent orchestrates their lifecycle; a mechanical graduation gate promotes them only on typed, committed evidence.

Human-in-the-loop by design. QB never writes code itself — it classifies, gates, and routes, and a deterministic Python driver (pipelines.yaml + externalized run-state) enforces phase order and refuses illegal transitions. Both checkpoints are structured questions; chat text never counts as approval, and hard gates (new Azure cost, auth, push, destructive ops) hold at every autonomy level.

The improvement system & eval harness

A closed loop: retro discovers → IMP files capture → evals measure → changes ship → next retro verifies impact.

Tracked improvements (IMPs)

The fleet evolves through one file per proposal — 60+ filed to date — moving through a proposed → accepted → implemented → validated lifecycle. Graduation to validated is a mechanical gate: typed, source-tagged evidence (deterministic · synthetic · surrogate · real-session · inspection) with commit provenance, Wilson-bound runtime confidence, ticked acceptance criteria, and a real CHANGELOG SHA.

  • All improvements (IMP backlog)
  • Conventions, lifecycle & the validated bar
  • Execution order — pain-point waves
  • Shipped changes log

The eval harness

A Python harness with nine eval types (structural, tool-loop, routing, behavioral, quality, rubric-judged, execution-metrics, composite, manual). A structural fleet gate runs nightly in CI; telemetry mining scores real sessions from both runtimes; run records feed completion / cycle-time / bounce KPIs.

  • evals/ overview
  • EVAL-SYSTEM-PLAN — the full design
  • Artifact schemas (run-state, brief, design preview…)

How it has evolved

Shipped in waves, each attacking a measured pain point:

  • Run-state + driver — externalized pipeline state, artifact-by-reference handoffs, a driver that refuses illegal transitions
  • Customer context — researched briefs with provenance + untrusted-content hardening
  • SCOUT + design preview — cheap recon before a mandatory CHECKPOINT 2 plan
  • Worktree isolation — per-track worktrees, attributed merges, checkpoint/rewind
  • Risk-tiered autonomy — session dial; hard gates never move
  • Synthetic-first graduation — the mechanical validated bar
  • In flight — DEV throughput (parallel track waves, rigor dial, deploy preflight) and cloud delegation to async coding agents

Operating discipline

  • Nightly jobs — structural CI gate, evidence backfill, hygiene sweep, untracked-work scan, public-mirror drift
  • Run records — every pipeline run emits outcome telemetry (completion, cycle time, bounces, escalations)
  • Ask discipline — clarification ledger; decisions asked once, delivery contract pinned at approval
  • Second opinions — ORACLE consulted at conflicts, advisory-only
  • This site — published from the private working system through an allowlist + redaction + secret-scan pipeline; push is always human-approved

Get started

These are agent definitions, not a standalone app — they run inside GitHub Copilot (VS Code Copilot Chat and/or the Copilot CLI).

1 · Install the agents

# Clone the repo
git clone https://github.com/harryschaefer93/qb-agent-system
cd qb-agent-system

# Copilot CLI: copy CLI agents into ~/.copilot/agents/
#   scoper.md, retro.md, imp.md

# VS Code Copilot Chat: copy *.agent.md into your
# custom-agents dir, then pick QB in the agent selector

2 · Run the pipeline

# (Optional) drop a BRIEF.md at your workspace root,
# or let scoper generate one.

# Invoke QB with a request, e.g.
QB ▸ "Build a POC for a RAG chatbot on Azure."

# QB confirms scope (Checkpoint 1), runs QA,
# presents a plan (Checkpoint 2), then routes work.

3 · Run the eval harness (optional)

cd evals
pip install -e .

# Dry-run an agent's behavioral test cases
python -m runner.cli run-behavioral qb --dry-run

# Mine session telemetry for IMP evidence
python -m runner.telemetry scan --since 30d

Adapt before you run

  • Model IDs — the fleet runs a three-tier economy: Opus-class (claude-opus-4.8-1m) for judgment agents, Sonnet-class for volume work, Haiku-class for SCOUT recon, plus a cross-family model for ORACLE. Swap for models your license exposes.
  • ~/.copilot paths — retro and parts of evals assume the standard local Copilot session store (OS-agnostic).
  • MCP servers — any you don't have are ignored; trim the tools: list if you prefer.
  • FDPO conventions — Entra ID + RBAC, no API keys, disableLocalAuth: true. No change needed for FDPO tenants.
Note on examples. All customer names and identifiers in this repo (Contoso, Fabrikam, Woodgrove, Adatum…) are fictional placeholders. Nothing here contains real customer data.
