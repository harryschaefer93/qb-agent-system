# Rubric: Evidence-Backed Recommendation Quality

## Criterion: citation_presence (weight: 0.25)
Does the recommendation include at least one URL citation to an authoritative source (Microsoft Learn, official Azure documentation, or first-party vendor documentation)? A citation alone is necessary but not sufficient — the recommendation must point at something readers can verify. Blogs, StackOverflow answers, and unsourced model-knowledge claims score low. The recommended option's `description` field is the canonical citation site (per IMP-0020 acceptance criteria).

### Score Definitions
- **5**: Well-formed URL to `*.microsoft.com`, `learn.microsoft.com`, `*.azure.com`, or an official first-party vendor doc. Citation is parseable, the domain is authoritative, and the URL appears in the recommended option's description.
- **4**: Valid URL to an authoritative source, but placement is informal (in chat text rather than the option's description field).
- **3**: Citation present but to a secondary source (Microsoft TechCommunity blog, official MS GitHub samples repo, well-known engineering blogs like the Azure Architecture Center community posts).
- **2**: Citation to a non-authoritative source (third-party blog, StackOverflow, Medium post) used as the primary basis for the recommendation.
- **1**: No citation present, OR citation is to a URL that does not resolve / is malformed / is a placeholder.

### Examples
- **5**: "Recommend Cosmos DB for NoSQL. Per [Microsoft Learn — Cosmos DB for AI applications](https://learn.microsoft.com/en-us/azure/cosmos-db/nosql/ai-advantage), it provides single-digit-ms reads at any scale ..."
- **3**: "Recommend Cosmos DB. The Azure Architecture Center has a good comparison post on this: <https://techcommunity.microsoft.com/...>."
- **1**: "Use Cosmos DB for chat history. It's designed for high-throughput workloads."

## Criterion: source_recommendation_alignment (weight: 0.35)
Do the claims in the recommendation align with the cited source as evidenced by the response text itself? This is the anti-hallucination gate. **The judge does NOT fetch URLs** — alignment is verified by examining the response: does it quote / paraphrase / cite specific passages from the source, and do those quoted passages support the recommendation's claims? A recommendation that cites a real Microsoft Learn page but then asserts the opposite of what the page says (e.g., "Cosmos DB has no SLA per [Microsoft Learn URL]" when the SLA is well-known) is a worst-case failure — it weaponizes the citation to give false authority to a wrong claim. Per MS Multi-Agent Reference Architecture: *"Detect when an agent invents citations or misrepresents known facts."*

### Score Definitions
- **5**: Recommendation includes verbatim quotes or specific paraphrases from the cited source that directly support every material claim. The judge can trace each claim back to evidence presented in the response itself.
- **4**: Most material claims supported by quoted / paraphrased source content; at most one minor inference reasonable from the quoted material.
- **3**: Some material claims have visible source-support in the response; one or two extensions go beyond what's quoted but don't contradict it.
- **2**: Significant material claim has no quoted source-support in the response — the citation URL is "decoration" while the actual recommendation content reads as model-knowledge.
- **1**: Recommendation directly contradicts a well-known fact from the cited source (e.g., claims "no SLA" when source is famous for documenting "99.999%"). Hard fail — this is the "fabricated authority" anti-pattern.

### Examples
- **5**: Cites `learn.microsoft.com/azure/cosmos-db/nosql/ai-advantage` and includes verbatim quotes: "'Azure Cosmos DB for NoSQL offers single-digit-millisecond response times'" + "'99.999% availability SLA for multi-region accounts'" + "'Native vector search using DiskANN'".
- **3**: Cites the ARM Templates overview page and recommends Bicep based on it. Quoted ARM-templates passage supports declarative-IaC framing but doesn't actually compare to Bicep — alignment is partial.
- **1**: Cites `learn.microsoft.com/azure/cosmos-db/high-availability` and claims "Cosmos DB has no guaranteed SLA for multi-region writes" — direct contradiction of well-known Cosmos DB SLA documentation.

## Criterion: context_relevance (weight: 0.25)
Does the recommendation directly address the specific decision question posed at the checkpoint, in the customer's actual context (Azure / FSI / FDPO-compliant)? A correct recommendation for a different question scores poorly. Per OpenEvals `RAG_RETRIEVAL_RELEVANCE_PROMPT` and Foundry `Relevance` evaluator.

### Score Definitions
- **5**: Recommendation directly and fully addresses the posed question, scoped to the actual context (e.g., "Cosmos DB vs PostgreSQL for chat history *on Foundry* in *FSI*"). Cited source is on-topic for the exact context.
- **4**: Addresses the question; minor scope mismatch (e.g., answers for general Azure rather than the specific Foundry-on-FSI context).
- **3**: Mostly relevant; addresses a parent or adjacent question rather than the specific one asked.
- **2**: Partially relevant — touches on the right topic but doesn't actually answer the decision question.
- **1**: Off-topic; the recommendation answers a different question, OR the cited source is unrelated to the recommendation content.

### Examples
- **5**: Question is "Cosmos DB vs PostgreSQL for chat history on Foundry FSI"; recommendation explicitly compares the two for that workload pattern, citing FSI-relevant constraints (FDPO, regulatory) where applicable.
- **3**: Same question; recommendation says "Use Cosmos DB" with a generic citation about Azure database choice that doesn't address chat-history-on-Foundry specifically.
- **1**: Question is "Cosmos DB vs PostgreSQL"; recommendation discusses Azure SQL vs Synapse and never answers the actual question.

## Criterion: recommendation_completeness (weight: 0.15)
Does the recommendation surface the material trade-offs and conditions documented in the cited source? Where the source discusses alternatives, prerequisites, or constraints, a complete recommendation acknowledges them. Per Foundry `Response Completeness` (recall dimension). Lowest weight because completeness is a "polish" criterion compared to correctness (alignment) and presence (citation).

### Score Definitions
- **5**: Recommendation comprehensively covers the trade-offs the source discusses (e.g., when recommending SAMI, mentions UAMI for multi-resource scenarios; when recommending Bicep, mentions Terraform's multi-cloud portability). No material omissions.
- **4**: Most trade-offs covered; one minor consideration from the source is omitted.
- **3**: Some trade-offs covered but a notable alternative or condition from the source is missing.
- **2**: Trade-offs largely omitted; recommendation reads as if the cited source had only the recommended option.
- **1**: Recommendation completely omits material trade-offs or constraints from the source. Reader would be surprised by limitations later.

### Examples
- **5**: SAMI recommendation explicitly contrasts with UAMI ("For multi-resource scenarios where the identity must span App Service + Functions + Container App, UAMI is the better choice").
- **3**: SAMI recommendation mentions managed identities in general but doesn't distinguish SAMI vs UAMI even though the source explicitly does.
- **1**: SAMI recommendation presents managed identities as the only option, omitting Workload Identity Federation entirely even though the source devotes a section to it.
